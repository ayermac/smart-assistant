# Plan 10-01: Obsidian 解析器与多模态嵌入

## Objective

实现 Obsidian vault 集成的核心解析能力：双向链接解析、图片路径解析、多模态嵌入客户端，以及数据结构扩展。

## Tasks

### Task 1: 扩展 KnowledgeChunk 类型

<read_first>
- src/knowledge/types.ts
- src/knowledge/vector-store.ts
</read_first>

<action>
在 `src/knowledge/types.ts` 中：

1. 新增 `ImageReference` 接口：
   - `path: string` — 图片绝对路径
   - `relativePath: string` — Markdown 中的相对路径
   - `altText?: string` — alt 文本

2. 扩展 `KnowledgeChunk` 接口，添加可选字段：
   - `images?: ImageReference[]` — 块中引用的图片

   - `linkedNotes?: string[]` — 双向链接目标笔记名
   - `lastModified?: number` — 文件最后修改时间戳（增量更新用）
   - `tags?: string[]` — Obsidian 标签（从 frontmatter 和 #tag 提取）

3. 确保 `ChunkMetadata` 包含 `linkedNotes` 和 `tags` 字段
</action>

- `npm run typecheck` 通过
</acceptance_criteria>

### Task 2: 实现 Obsidian 解析器

<read_first>
- src/knowledge/types.ts
- src/knowledge/chunker.ts
- src/knowledge/cleaner.ts
</read_first>

<action>
创建 `src/knowledge/obsidian.ts`，实现：

1. `parseWikiLinks(content: string): string[]` — 解析 `[[note-name]]` 双向链接
   - 支持 `[[note-name]]` 和 `[[note-name|display text]]` 两种格式
   - 返回去重的笔记名列表

2. `parseImages(content: string, vaultPath: string): ImageReference[]` — 解析图片引用
   - 匹配 `![alt](path)` 格式
   - 将相对路径结合 `vaultPath` 解析为绝对路径
   - 返回 `ImageReference[]`

3. `parseTags(content: string): string[]` — 解析 Obsidian 标签
   - 匹配 `#tag` 格式（排除代码块内的标签）
   - 匹配 frontmatter 中的 `tags: [tag1, tag2]`
   - 返回去重标签列表

4. `resolveNotePath(noteName: string, vaultPath: string): string | null` — 解析笔记路径
   - 尝试 `${vaultPath}/${noteName}.md`
   - 尝试 `${vaultPath}/${noteName}.markdown`
   - 未找到返回 null
</action>

<acceptance_criteria>
- `src/knowledge/obsidian.ts` 存在并导出 `parseWikiLinks`, `parseImages`, `parseTags`, `resolveNotePath`
- `parseWikiLinks("见 [[API设计]] 和 [[数据库|DB]]")` 返回 `["API设计", "数据库"]`
- `parseImages("![img](attachments/photo.png)", "/vault")` 返回包含 `path: "/vault/attachments/photo.png"` 的数组
- `parseTags("#rust #typescript")` 返回 `["rust", "typescript"]`
- `npm run typecheck` 通过
</acceptance_criteria>

### Task 3: 实现多模态嵌入客户端

<read_first>
- src/memory/embedding.ts
- src/knowledge/types.ts
</read_first>

<action>
创建 `src/knowledge/multimodal-embedding.ts`，实现：

1. `MultimodalInput` 类型：
   ```typescript
   type MultimodalInput = {
     text?: string;
     image?: string; // Base64 编码或可公网访问的 URL
   };
   ```

2. `getMultimodalEmbedding(input: MultimodalInput, config: EmbeddingConfig): Promise<number[]>`
   - 构建与 doubao-embedding-vision 兼容的请求体
   - 250615+ 版本：`input` 字段支持 `type: "text"` 和 `type: "image_url"` 混排
   - 单文本输入时保持与现有 `getEmbedding()` 相同的请求格式（向后兼容）
   - 有图片输入时构建多模态请求体
   - 返回 2048 维向量

3. `imageToBase64(imagePath: string): Promise<string>` — 读取图片文件并转为 Base64
   - 支持 PNG, JPG, JPEG, GIF, WebP 格式
   - 超过 4MB 的图片跳过并返回空字符串（避免 API 限制）
</action>

<acceptance_criteria>
- `src/knowledge/multimodal-embedding.ts` 存在并导出 `getMultimodalEmbedding`, `imageToBase64`, `MultimodalInput`
- 纯文本调用 `getMultimodalEmbedding({ text: "hello" }, config)` 返回 2048 维向量
- 图文混合调用 `getMultimodalEmbedding({ text: "描述", image: "data:image/png;base64,..." }, config)` 返回 2048 维向量
- `imageToBase64` 对不存在的文件抛出错误
- `npm run typecheck` 通过
</acceptance_criteria>
