# Plan 05-02: Markdown/Text Chunking and Local Index Storage

**Goal:** Implement Markdown heading-based chunking and FileKnowledgeStore with JSON file storage and manifest.

**Requirements:** RAG-02, RAG-06 (partial)

## Tasks

### Task 1: Implement Markdown/text chunker

<read_first>
- src/knowledge/types.ts
- src/config.ts
</read_first>

<action>
Create src/knowledge/chunker.ts. Export chunkFile function: (filePath: string, content: string) => KnowledgeChunk[]. Detect file type by extension (.md, .markdown → Markdown; .txt → plain text). For Markdown: parse lines, detect heading pattern (regex /^#{1,6}\s+/), each heading starts a new chunk, content before first heading becomes a chunk with empty headingText and headingLevel 0. For plain text: return single chunk with headingLevel 0. Each chunk gets a UUID id, sourcePath as relative path, startLine/endLine from parsing, tags as empty array, createdAt as ISO timestamp. Export SUPPORTED_EXTENSIONS constant.
</action>

<acceptance_criteria>
- src/knowledge/chunker.ts exports chunkFile function
- chunkFile returns KnowledgeChunk[] for supported extensions
- Markdown files split at heading boundaries (#, ##, ###)
- Content before first heading becomes chunk with empty headingText
- Plain text files (.txt) return single chunk with headingLevel 0
- Each chunk includes sourcePath, headingLevel, headingText, text, startLine, endLine, id, createdAt
- Empty content returns empty array
- Unsupported extensions return empty array
- pnpm tsc --noEmit passes
</acceptance_criteria>

### Task 2: Implement FileKnowledgeStore with ingestion and manifest

<read_first>
- src/memory/store.ts (pattern reference for atomicWriteJson, file structure)
- src/knowledge/types.ts
- src/knowledge/chunker.ts
- src/config.ts
</read_first>

<action>
Create src/knowledge/store.ts. Implement FileKnowledgeStore class implementing KnowledgeStore. Constructor takes optional config (knowledgeDir, sourceDir paths), defaults from resolveDataPaths().knowledge and resolveKnowledgeSourceDir(). Implement atomicWriteJson following FileMemoryStore pattern (write .tmp then rename). Implement ingest: scan sourceDir recursively for supported extensions, chunk each file via chunkFile, write each chunk to {knowledgeDir}/chunks/chunk-{id}.json, build and write manifest.json to {knowledgeDir}/manifest.json. Skip files that fail to read (log warning). Skip hidden files (starting with .). Implement getManifest: read and parse manifest.json. Implement getChunk: read chunk file by id. Implement listChunks: read manifest, load all chunk files.
</action>

<acceptance_criteria>
- src/knowledge/store.ts exports FileKnowledgeStore class implementing KnowledgeStore
- ingest scans sourceDir recursively for .md, .txt, .markdown files
- Chunks written to {knowledgeDir}/chunks/chunk-{id}.json with atomic writes
- manifest.json written to {knowledgeDir}/manifest.json after ingestion
- Manifest includes version, lastIndexed (ISO 8601), sourceDir, chunks metadata, sources with mtime
- getManifest returns KnowledgeManifest or null if not ingested
- getChunk returns KnowledgeChunk by id or null
- listChunks returns all chunks
- Hidden files and unsupported extensions are skipped
- Read errors on individual files are logged and skipped (not thrown)
- pnpm tsc --noEmit passes
</acceptance_criteria>

### Task 3: Implement needsReindex and search in FileKnowledgeStore

<read_first>
- src/memory/store.ts (recall method pattern for scoring)
- src/knowledge/types.ts
- src/knowledge/store.ts
</read_first>

<action>
In FileKnowledgeStore, implement needsReindex: return true if manifest is null, if any source file mtime changed, or if new files exist not in manifest. Implement search: call needsReindex and ingest if needed, then load all chunks via listChunks, score each chunk with keyword matching (substring match +10, word overlap +1 per word, heading match +5, tag overlap +2 per tag, source path match +3). Sort by relevanceScore desc, apply optional tags filter, sourcePath filter, and limit (default 5). Return KnowledgeMatch[] with chunk, score, and matchReason.
</action>

<acceptance_criteria>

- search triggers ingestion if needsReindex returns true
- search returns KnowledgeMatch[] sorted by relevanceScore descending
- search respects limit option (default 5)
- search supports optional tags filter
- search supports optional sourcePath filter
- Relevance scoring uses: substring +10, word overlap +1/word, heading +5, tag +2/tag, source path +3
- Empty query returns empty array
- pnpm tsc --noEmit passes
</acceptance_criteria>
