# 14-01 Plan: Stabilize Obsidian Incremental Startup Sync

## Context

TUI startup logs showed many Obsidian vault files being reindexed on every launch. The intended behavior in `VectorKnowledgeStore.syncVault()` is incremental startup sync: scan current files, compare stored file mtimes, index only new files, reindex modified files, and remove deleted files.

The likely failure mode is stale local LanceDB metadata. Older rows may have `lastModified` stored with an incompatible integer representation while the code compared it against JavaScript `stat().mtimeMs` millisecond values. Those rows look modified on every startup even when the source files did not change.

## Goals

- Preserve CLI/TUI startup behavior and continue calling `syncVault()` when `OBSIDIAN_VAULT_PATH` is configured.
- Store reliable millisecond file mtimes for future incremental comparisons.
- Treat unusable legacy mtimes as a one-time reindex trigger so existing local databases repair themselves.
- Keep docs under `.planning` and update README verification guidance.
- Add focused tests for mtime compatibility helpers.

## Implementation Steps

1. Add a `lastModifiedMs` Float64 column to new and migrated knowledge tables.
2. Keep `lastModified` for compatibility, but write seconds into it so older Int32 schemas no longer truncate millisecond values.
3. Read stored mtimes through a helper that prefers `lastModifiedMs`, accepts short-lived Float64 millisecond `lastModified` rows, and rejects old truncated values.
4. Make vault sync reindex rows with unusable stored mtimes once, then write back `lastModifiedMs`.
5. Add tests covering helper behavior and legacy timestamp rejection.
6. Update README and README_CN with incremental startup verification steps.
7. Run typecheck, tests, and build.

## Acceptance Criteria

- `syncVault()` remains incremental for unchanged files once reliable mtimes are stored.
- Legacy rows with unusable mtimes are repaired by reindexing once.
- `npm run typecheck`, `npm test`, and `npm run build` pass.
- User-facing docs explain why the first run after upgrade may reindex and how to verify the second run is a no-op.
