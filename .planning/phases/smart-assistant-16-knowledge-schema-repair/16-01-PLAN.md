# 16-01 Plan: Repair Corrupted Knowledge Mtime Schema

## Context

After dependency and mtime fixes, TUI startup still reported:

`Failed to set up vault watching: Panic in async function`

Directly running `VectorKnowledgeStore.syncVault()` showed the LanceDB panic:

`Column 'lastModifiedMs' is declared as non-nullable but contains null values`

The local knowledge table had an old `lastModified` Int32 column and a newly added non-nullable `lastModifiedMs` column. Selecting that column caused the LanceDB Rust layer to panic before vault sync could complete.

## Goals

- Avoid LanceDB panics caused by incompatible legacy knowledge table schemas.
- Detect old mtime schemas during `VectorKnowledgeStore.init()`.
- Recreate only the `knowledge` table when mtime columns are incompatible; do not touch the `memories` table.
- Let the next vault sync rebuild the knowledge index from source files.
- Add tests for mtime schema compatibility detection.

## Implementation Steps

1. Extract a single `createKnowledgeTableSchema()` helper for the canonical knowledge table schema.
2. Add `hasCompatibleKnowledgeMtimeSchema()` to validate `lastModified` and `lastModifiedMs`.
3. Treat missing, non-Float64, or non-nullable mtime fields as incompatible.
4. Recreate the `knowledge` table when incompatible mtime schema is detected.
5. Add unit tests covering compatible, Int32 legacy, non-nullable, and missing mtime schemas.
6. Verify with a temporary LanceDB table that mirrors the broken local schema.
7. Run typecheck, tests, build, and diff checks.

## Acceptance Criteria

- `VectorKnowledgeStore.init()` repairs bad mtime schema before `syncVault()` queries `lastModifiedMs`.
- TUI startup should no longer fail with `Panic in async function` for this schema state.
- First startup after repair may rebuild the knowledge index once.
- `npm run typecheck`, `npm test`, and `npm run build` pass.
