# 17-01 Plan: Fix TUI Input and Exit During Initialization

## Context

The TUI displayed the input prompt while still showing `Initializing assistant...`, but typed input did not behave reliably. The user could not exit or submit text from the visible `you>` prompt.

The implementation discarded all non-Ctrl+C input while `isInitializing` was true. In some terminal/PTY modes, pressing Enter also arrived as a raw newline character instead of `key.return`, so `/exit` could appear in the input box without being submitted.

## Goals

- Allow `/exit` and Ctrl+C to work even while the assistant runtime is still initializing.
- Allow typed characters to appear in the input box during initialization.
- Treat raw `\n` and `\r` as submit signals in addition to Ink's `key.return`.
- Ensure explicit TUI exit terminates the Node process instead of leaving background initialization work alive.
- Preserve active-response Ctrl+C behavior: abort response first, exit when idle.

## Implementation Steps

1. Stop discarding input solely because `isInitializing` is true.
2. Keep prompt-in-progress input blocking so active assistant responses remain protected.
3. Add a shared interrupt handler used by both Ink `Ctrl+C` input and process `SIGINT`.
4. Track prompt state in a ref so SIGINT can see the latest state.
5. Make `/exit` and empty submit call a guarded `stopAndExit()`.
6. Make `stopAndExit()` call Ink `exit()` and then `process.exit(0)`.
7. Support newline/carriage-return values as submit triggers.
8. Verify with an interactive temporary TUI session by sending `/exit\n`.

## Acceptance Criteria

- The user can type into the visible prompt during initialization.
- `/exit` exits from the initialization screen.
- Ctrl+C exits when idle and aborts only when a response is active.
- `npm run typecheck`, `npm test`, and `npm run build` pass.
