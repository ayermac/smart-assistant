# 21-01 Plan: Operability and RAG Stability Hardening

## Objective

Improve the assistant's operational reliability by adding diagnostics, serializing knowledge-store writes, and widening type/test coverage around the RAG path.

## User Story

As the project maintainer, I want slow or stuck knowledge behavior to be observable and reproducible, so I can trust the assistant during daily TUI usage and debug failures without reading LanceDB internals.

## Design Principles

- Keep implementation local-first and dependency-light.
- Prefer small composable helpers over a heavy observability framework.
- Preserve current CLI/TUI behavior unless diagnostics are explicitly enabled.
- Avoid changing retrieval semantics while hardening execution.
- Keep planning and verification artifacts under `.planning`.

## Workstreams

### 1. Structured Local Logger

Add a small logger module used by runtime, watcher, and knowledge code.

Tasks:

- Add `src/logger.ts` with levels: `silent`, `error`, `warn`, `info`, `debug`.
- Read `SMART_ASSISTANT_LOG_LEVEL`, defaulting to `info`.
- Support optional timing metadata in a stable single-line format.
- Replace high-value `console.warn/error/log` calls in RAG startup, watcher, and tool paths.
- Keep user-facing CLI/TUI messages separate from debug logs.

Acceptance criteria:

- `SMART_ASSISTANT_LOG_LEVEL=debug` emits RAG stage details.
- Default output remains close to current behavior.
- Tests cover level filtering and metadata formatting.

### 2. RAG Timing Instrumentation

Measure the expensive RAG stages without changing retrieval behavior.

Tasks:

- Add a timing helper around:
  - `needsReindex`
  - `syncVault`
  - `ingest`
  - `indexFile`
  - query embedding
  - LanceDB vector search
  - BM25 rebuild/search
  - RRF fusion
  - rerank
- Include counts where available: files scanned, chunks indexed, vector matches, BM25 matches, final matches.
- Surface concise tool progress in TUI/CLI where useful.

Acceptance criteria:

- A slow `search_knowledge` request can be traced stage by stage in debug logs.
- Unit tests verify timing wrapper emits duration and preserves thrown errors.
- No timing logs appear at `silent` or below configured level.

### 3. Knowledge Write Serialization

Prevent overlapping writes to the LanceDB knowledge table.

Tasks:

- Add a lightweight async queue/mutex for write operations.
- Serialize:
  - `ingest`
  - `indexFile`
  - `reindexFile`
  - `removeFile`
  - `syncVault`
  - schema migration/recreate operations if practical
- Allow searches to wait for an active write only when needed, or run after the current write completes if schema/table state may be unstable.
- Add debug logs when operations wait behind a write.

Acceptance criteria:

- Concurrent watcher updates and search-triggered reindex do not run table writes in parallel.
- Tests simulate concurrent write calls and prove execution order.
- Abort/timeout behavior from `search_knowledge` is preserved.

### 4. Script Typechecking

Make script drift visible in normal verification.

Tasks:

- Add `tsconfig.scripts.json` or adjust project references so `scripts/*.ts` typecheck.
- Add `npm run typecheck:scripts`.
- Add `npm run verify` to run typecheck, script typecheck, tests, and build.
- Fix script API mismatches discovered by the new check.

Acceptance criteria:

- `npm run typecheck:scripts` passes.
- `npm run verify` passes locally.
- README/README_CN document the new verification command.

### 5. End-to-End RAG Regression Test

Add one focused integration test for the actual indexing/search path.

Tasks:

- Use a temporary knowledge directory and data directory.
- Mock embedding with deterministic vectors.
- Index a small Markdown corpus.
- Run `search_knowledge` through the tool or `VectorKnowledgeStore.search`.
- Assert source path, citation details, and non-empty match.

Acceptance criteria:

- Test does not require network.
- Test does not write outside temp directories.
- Test proves index + search + source metadata work together.

### 6. Documentation Update

Document the operational model.

Tasks:

- README/README_CN:
  - Add `SMART_ASSISTANT_LOG_LEVEL`
  - Add `npm run verify`
  - Add short diagnostic flow for slow `search_knowledge`
- CHANGELOG:
  - Add Unreleased entries.
- `.planning`:
  - Add implementation summary when complete.

Acceptance criteria:

- A maintainer can follow docs to debug a slow RAG request.
- Verification commands are explicit and current.

## Suggested Implementation Order

1. Add logger and timing helper with tests.
2. Add RAG timing logs around read-only stages.
3. Add write serialization for mutation paths.
4. Add script typecheck and fix script drift.
5. Add E2E RAG regression test.
6. Update docs and run full verification.

## Verification Plan

Required commands:

```bash
npm run typecheck
npm run typecheck:scripts
npm test
npm run build
node dist/cli.js --help
node dist/tui.js --help
```

If `npm run verify` is added, it should run the first four commands.

Manual checks:

- Start TUI with `SMART_ASSISTANT_LOG_LEVEL=debug`.
- Ask a question that triggers `search_knowledge`.
- Confirm progress appears in TUI and timing details appear in logs.
- Modify one watched Markdown file and verify watcher indexing does not overlap with a search-triggered write.

## Risks

- Over-serializing reads could make search feel slower.
- Debug logs could pollute TUI output if stdout/stderr routing is not kept clean.
- E2E LanceDB tests may be sensitive to table schema setup and cleanup.
- Script typechecking may expose unrelated drift; keep fixes minimal.

## Non-Goals

- Ranking quality improvements.
- Query rewriting.
- Persistent BM25 storage.
- Web UI or Gradio UI.
- Packaging or publishing to npm registry.

## Definition of Done

- Logger and timing helper are tested.
- Knowledge writes are serialized.
- Script typecheck is part of documented verification.
- RAG integration test runs offline.
- README, README_CN, CHANGELOG, and `.planning` are updated.
- Full verification passes.
